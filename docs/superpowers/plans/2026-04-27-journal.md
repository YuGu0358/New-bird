# Investment Journal — Daily Research Notes Linked to Context

**Status:** Pending. To be executed via `/subagent-driven-development`.

**Goal:** Give the user a single place to write daily investment notes that auto-link to the rest of Newbird's data — symbol mentions become clickable, AI Council analyses can be referenced, risk events can be cited, social-signal snapshots can be quoted. The journal is the "why I'm in this trade" layer that ties everything else together.

**Why this is shippable:**
- No external API keys, no third-party services, no broker
- Pure CRUD against a single SQLite table
- Front-end fits naturally next to Markets / Portfolio / Intelligence
- 5 small tasks, each one isolated enough for a subagent to handle in a single dispatch

---

## Architecture

```
┌──────────────────────────────────────────────────────────────────┐
│  /journal                                                        │
│  ┌─ entry list (chronological, paginated, searchable)         │
│  │   [date]  [tags]  [title]  [first 80 chars of body]        │
│  │                                                             │
│  └─ "+ New Entry" button → opens editor modal                 │
│                                                                  │
│  Editor modal:                                                   │
│   ├─ Title (single line)                                        │
│   ├─ Body (markdown textarea — no rich editor, KISS)           │
│   ├─ Symbol tags (comma-separated, autocompletes from           │
│   │   monitoring + watchlist)                                   │
│   ├─ Mood (bullish | bearish | neutral | watching)              │
│   ├─ Save / Cancel                                              │
└──────────────────────────────────────────────────────────────────┘
                                │
                                ▼
              ┌───────────────────────────────────┐
              │  POST /api/journal                │
              │  GET  /api/journal                │
              │  GET  /api/journal/{id}           │
              │  PATCH /api/journal/{id}          │
              │  DELETE /api/journal/{id}         │
              └───────────────┬───────────────────┘
                              ▼
              ┌───────────────────────────────────┐
              │  journal_entries (SQLite)         │
              │   id, title, body, symbols (JSON),│
              │   mood, created_at, updated_at    │
              └───────────────────────────────────┘
```

**Out of scope (deliberate):**
- Rich-text / WYSIWYG editor (textarea is fine — markdown rendered on view)
- Linking entries to specific AgentAnalysis / RiskEvent rows by FK (Phase 2 — for now we just store free-text symbol tags; linking is a future enhancement)
- Backlinks / wikilinks across entries
- Image / file attachments
- Full-text search (substring `LIKE` filter is enough for MVP)
- Public sharing / multi-user

**Tech stack:** SQLAlchemy table + FastAPI router + React page. Same patterns as the existing `alerts` and `agents/history` features — copy from there, no new deps.

**Spec:**

`JournalEntry` columns:
```
id            int, primary key, auto-increment, indexed
title         str(120), not null, default ""
body          Text, not null, default ""           # markdown
symbols       JSON, not null, default []           # list[str], all uppercased
mood          str(16), not null, default "neutral" # enum: bullish|bearish|neutral|watching
created_at    DateTime(tz=UTC), not null, default now(), indexed (DESC for sort)
updated_at    DateTime(tz=UTC), not null, default now(), onupdate=now()
```

`mood` is stored as a plain string (no DB-level enum constraint — the service-layer validator enforces it). Consistent with how `RiskEvent.action` etc. work in this codebase.

**Test (write first):**

## Task 2: Service layer

**Files:**
- Create: `backend/app/services/journal_service.py`
- Create: `backend/tests/test_journal_service.py`

**Public surface:**

```python
async def list_entries(
    session,
    *,
    symbol: str | None = None,
    mood: str | None = None,
    search: str | None = None,
    limit: int = 50,
    offset: int = 0,
) -> list[dict[str, Any]]:
    """List entries newest-first. Filters compose with AND.

    - symbol: filter to entries that mention this symbol (JSON contains)
    - mood: filter to one mood value
    - search: substring match (case-insensitive) on title OR body
    - limit/offset: pagination, limit clamped to [1, 200]
    """

async def get_entry(session, entry_id: int) -> dict[str, Any] | None:
    """Fetch one entry by id. None if missing."""

async def create_entry(
    session,
    *,
    title: str,
    body: str,
    symbols: list[str] | None = None,
    mood: str = "neutral",
) -> dict[str, Any]:
    """Create. Title trimmed, capped at 120 chars. Body length unbounded.
    Symbols normalized to upper-case + de-duplicated. mood validated against
    the allowed set ('bullish'|'bearish'|'neutral'|'watching')."""

async def update_entry(
    session, entry_id: int, *,
    title: str | None = None,
    body: str | None = None,
    symbols: list[str] | None = None,
    mood: str | None = None,
) -> dict[str, Any] | None:
    """Patch the fields that are not None. None on missing entry. Validation
    matches create_entry."""

async def delete_entry(session, entry_id: int) -> bool:
    """True if a row was removed, False if it didn't exist."""

async def autocomplete_symbols(session, *, prefix: str, limit: int = 10) -> list[str]:
    """Return distinct symbols across all entries that start with `prefix`
    (case-insensitive). Used by the symbol-tag autocomplete in the editor."""
```

**Validation rules (must be tested):**
- `mood` not in the allowed set → `ValueError`
- `title` is required and non-empty after `.strip()` (otherwise `ValueError`); UI is allowed to show "Untitled" but the API must require it
- `symbols` deduplicated and uppercased; rejects entries with > 20 symbols (sanity cap)
- `search` substring match is case-insensitive (use SQLAlchemy `func.lower()`)

**Tests (TDD — write first, all of them):**
- create + list returns the entry
- list newest-first ordering
- list filtered by symbol returns only matching entries (JSON contains works in SQLite via `LIKE` on the serialized string — check the existing `RiskEvent.context_json` patterns first to see how others do JSON containment in this codebase; if there's no helper, use `WHERE symbols LIKE :pat` with `%"NVDA"%` literal — the test must verify it doesn't false-match `"NVDA-Inverse"` or similar)
- list filtered by mood works
- list search matches title and body
- limit/offset paginates
- update changes only the fields passed
- delete + 404 when re-fetching
- mood validation rejects bad values
- autocomplete returns distinct upper-cased symbols matching prefix

**Acceptance:** All tests pass. No code outside `journal_service.py` and `tests/test_journal_service.py` modified.

**Pydantic models:**

```python
class JournalEntryView(BaseModel):
    id: int
    title: str
    body: str
    symbols: list[str]
    mood: Literal["bullish", "bearish", "neutral", "watching"]
    created_at: datetime
    updated_at: datetime

class JournalEntryCreateRequest(BaseModel):
    title: str
    body: str = ""
    symbols: list[str] = []
    mood: Literal["bullish", "bearish", "neutral", "watching"] = "neutral"

class JournalEntryUpdateRequest(BaseModel):
    # all optional — PATCH semantics
    title: Optional[str] = None
    body: Optional[str] = None
    symbols: Optional[list[str]] = None
    mood: Optional[Literal["bullish", "bearish", "neutral", "watching"]] = None

class JournalListResponse(BaseModel):
    items: list[JournalEntryView]
    total: int           # total matching the filters (for pagination UI)
    limit: int
    offset: int

class JournalAutocompleteResponse(BaseModel):
    symbols: list[str]
```

**Endpoints:**

| Method | Path | Body | Response |
|---|---|---|---|
| GET | `/api/journal` | — (query params: symbol, mood, search, limit, offset) | `JournalListResponse` |
| POST | `/api/journal` | `JournalEntryCreateRequest` | `JournalEntryView` |
| GET | `/api/journal/{id}` | — | `JournalEntryView` (404 if missing) |
| PATCH | `/api/journal/{id}` | `JournalEntryUpdateRequest` | `JournalEntryView` (404 if missing) |
| DELETE | `/api/journal/{id}` | — | `{"removed": bool}` |
| GET | `/api/journal/symbols/autocomplete` | — (query: `prefix`, `limit`) | `JournalAutocompleteResponse` |

(So that's 6 routes, not 5 — autocomplete became its own thing for clarity. Update the parity test accordingly.)
